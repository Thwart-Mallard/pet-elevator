# Software Architecture

## Overview

The software is split across three Python packages:

- **`controller/`** — runs on the Raspberry Pi 4B (fixed, mains-powered). Owns
  the stepper motor, reads safety switches, maintains the elevator state machine,
  runs an MQTT broker (Mosquitto), and serves a local web UI for manual control.

- **`camera_node/`** — runs on each of the two landing Pi Zero 2 W units (fixed,
  mains-powered, one per floor). Captures frames from the CSI camera, runs
  on-device TFLite inference, and publishes dog detection events over WiFi.

- **`kart_node/`** — runs on the kart Pi Zero 2 W (solar + battery, no camera).
  Monitors a door NC switch and a pressure mat NO switch via GPIO and publishes
  their state as retained MQTT messages over WiFi.

## Controller Module Map

```
controller/main.py
  │  wires everything together, handles SIGTERM for clean shutdown
  │
  ├── config.py          GPIO pins, motion parameters, MQTT settings
  │
  ├── motor.py           StepperMotor
  │     pigpio DMA waveforms for accurate step timing
  │     trapezoidal velocity profile (accel → cruise loop → decel)
  │
  ├── safety.py          SafetyMonitor
  │     pigpio edge callbacks on all NC switch pins
  │     defers bottom-limit callback until after homing
  │
  ├── state_machine.py   ElevatorFSM
  │     owns all elevator logic
  │     spawns daemon threads for moves and homing
  │
  ├── mqtt_client.py     MQTTClient
  │     paho-mqtt wrapper
  │     subscribes to commands and camera detections
  │     publishes retained status on every state change
  │
  └── web_ui.py          WebUI
        Flask web server on port 8080
        SSE endpoint for real-time status push to browsers
        REST endpoints forward button presses to MQTT
        templates/index.html — mobile-friendly control UI
```

## motor.py — StepperMotor

### Step timing

Step pulses are generated by the `pigpio` DMA waveform engine — not by Python
timing loops. The DMA controller runs entirely in hardware, so timing is accurate
to ±1 µs regardless of Python's GIL or OS scheduling jitter.

### Velocity profile

Every move uses a trapezoidal profile computed before the wave is sent:

```
velocity
  │        ┌──────────────┐   ← v_peak (MAX_SPEED_STEPS_S or lower for short moves)
  │       /│              │\
  │      / │              │ \
  │     /  │              │  \
  │    /   │              │   \
  ────────────────────────────────► steps
       accel   cruise (looped)  decel
```

For short moves that cannot reach full speed (accel + decel > total steps), a
triangular profile is used automatically and v_peak is reduced accordingly.

### Wave construction

The cruise phase is implemented as a single one-step pigpio wave repeated N times
via `wave_chain` loop tokens. This handles the ~403,000 cruise steps in a
floor-to-floor move without creating 403,000 individual wave entries. The
`wave_chain` loop counter is 16-bit (max 65,535), so large cruise counts are
broken into multiple consecutive loop blocks automatically.

Acceleration and deceleration ramps are pre-computed as waves with per-step
variable intervals. If a ramp exceeds 10,000 pulses per wave (the conservative
pigpio limit) it is split into multiple waves automatically.

### Key methods

| Method | Description |
|--------|-------------|
| `home(safety_monitor)` | Crawl DOWN at homing speed until bottom limit switch opens. Sets `current_position = 0`. |
| `move_to_floor(floor)` | Move to absolute floor position. Blocks the caller thread until complete. |
| `request_stop()` | Thread-safe immediate stop — calls `pi.wave_tx_stop()`. |
| `enable()` / `disable()` | ENA+ LOW / HIGH. `disable()` is for maintenance only. |

## safety.py — SafetyMonitor

Uses `pigpio` `RISING_EDGE` callbacks — fires the fault handler the instant a NC
switch opens, with no polling delay.

**Bottom-limit callback deferral:** The bottom limit switch also fires during
homing (that is the intended trigger). The callback for GPIO 17 is not registered
until `enable_bottom_limit()` is called after homing succeeds. During homing the
motor polls the pin directly in its own loop.

The `suppress_bottom_limit` flag is set `True` for the duration of each homing
run (in case `enable_bottom_limit()` has been called on a re-home). The
motor's homing loop holds the RLock, so the callback and the polling loop cannot
race.

## state_machine.py — ElevatorFSM

### States

```
            power-on
               │
               ▼
            HOMING ──── homing failed ────────────────┐
               │                                       │
         homing success                                │
               │                                       ▼
            IDLE ◄──── move complete ────── MOVING    FAULT
               │                              ▲        │
               └──────── cmd_go() ────────────┘        │
                                                        │
         (any state) ──── safety fault ──────────────► │
                                                        │
                          cmd_reset() ──────────── HOMING
```

### Thread safety

All state transitions go through `_set_state()`, which acquires a `threading.RLock`
before writing `self.state`. `RLock` is used (rather than `Lock`) because
`_set_state` is called both from within `with self._lock` blocks (normal command
paths) and directly from the pigpio callback thread (`on_safety_fault`).

`on_status_change` is called **outside** the lock to avoid holding it during a
network publish.

### Key behaviours

**`cmd_go(floor)`:** Ignored unless state is `IDLE`. Transitions to `MOVING`,
then spawns a daemon thread that calls `motor.move_to_floor()`. On completion the
thread transitions to `IDLE` (success) or `FAULT` (failure or safety fault).

**`on_safety_fault(pin_name)`:** Called from the pigpio callback thread. Calls
`motor.request_stop()` immediately (stops DMA), then transitions to `FAULT`.
The moving thread wakes from its `wave_tx_busy` poll, sees `FAULT`, and exits
without overwriting the state.

**`cmd_reset()`:** Triggers re-homing. Only accepted from `IDLE` or `FAULT`
states. After successful re-homing the system returns to `IDLE` with
`current_floor = 0`.

## mqtt_client.py — MQTTClient

### Topics

| Direction | Topic | Payload |
|-----------|-------|---------|
| Subscribe | `elevator/command` | `{"action": "go", "floor": 0}` |
| Subscribe | `elevator/command` | `{"action": "home"}` |
| Subscribe | `elevator/command` | `{"action": "reset"}` |
| Subscribe | `elevator/camera/+/detection`    | `{"floor": 0, "confidence": 0.92}` |
| Subscribe | `elevator/kart/door`             | `{"status": "open"\|"closed"}` |
| Subscribe | `elevator/kart/pressure`         | `{"dog_present": true\|false}` |
| Publish   | `elevator/status` | `{"state": "idle", "floor": 0, "position": 0}` |

Status is published as **QoS 1, retained** so any new subscriber (phone, Home
Assistant, etc.) receives the current state immediately on connect.

### Detection handling

A camera detection event is acted on only if `confidence ≥ 0.80`. Below that
threshold the event is logged at DEBUG level and discarded. The FSM's `cmd_go()`
silently ignores the request if the elevator is already at that floor or in motion,
so repeated detections while the elevator travels are harmless.

### Status payload fields

| Field | Type | Description |
|-------|------|-------------|
| `state` | string | `boot`, `homing`, `idle`, `moving`, `fault` |
| `floor` | int \| null | `0` = ground, `1` = upper, `null` = in transit or unknown |
| `position` | int \| null | Steps from home (0 = ground floor) |
| `kart_door` | string | `open`, `closed`, `unknown` (unknown = kart node not yet connected) |
| `dog_present` | bool \| null | `true` = dog on pressure mat, `null` = kart node not yet connected |

## Camera Node Module Map

```
camera_node/main.py    CameraNode — capture loop
  │
  ├── config.py        floor identity (ELEVATOR_FLOOR env var), MQTT, model paths
  └── detector.py      DogDetector — TFLite MobileNet SSD wrapper
```

### Inference pipeline

```
picamera2.capture_array()          → H×W×3 uint8 RGB numpy array
PIL.Image.fromarray().resize(300×300)
numpy.expand_dims(..., axis=0)     → [1, 300, 300, 3] uint8 tensor
tflite.Interpreter.invoke()
get_tensor(classes), get_tensor(scores)
find max score where class == dog_class_id
publish if score ≥ MIN_SCORE (0.50)
```

Inference runs every `INFERENCE_INTERVAL` seconds (default 2.0). At ~1 fps
effective throughput on the Zero 2 W this is a realistic interval that avoids
overloading the CPU while still detecting a dog that has just approached the
landing.

### Model

MobileNet SSD v1 COCO quantised TFLite model (INT8). The COCO dataset includes
the "dog" class. The `DogDetector` finds the dog class index by searching the
label map file for the string "dog" rather than hardcoding an index, making it
robust to different COCO label file formats.

The model runs with `num_threads=4` to use all four cores on the Zero 2 W.

## web_ui.py — WebUI

### Architecture

The web UI runs as a Flask server (port 8080) in a daemon thread inside the main
controller process. It has its own independent paho-mqtt connection to the local
Mosquitto broker, keeping it fully decoupled from the controller's `MQTTClient`.

```
Browser ──── GET /          ──► Flask ──► render index.html
        ──── GET /api/events ──► Flask ──► SSE stream (per-tab queue)
        ──── POST /api/command ─► Flask ──► MQTT publish → elevator/command
                                               ↓ (picked up by MQTTClient)
                                           ElevatorFSM.cmd_*()

Mosquitto ── elevator/status ──► WebUI._on_message() ──► broadcast to all SSE queues
```

### Server-Sent Events (`/api/events`)

SSE gives real-time one-way push from server to browser without polling or
WebSockets. Each browser tab that opens the page gets its own `queue.Queue`.
When a status update arrives from MQTT, `WebUI._on_message` puts it into every
subscriber queue simultaneously. Each SSE generator picks the data off its queue
and yields it as an `event-stream` chunk.

A 25-second heartbeat comment (`": heartbeat"`) is yielded when the queue is
idle, preventing mobile browsers and reverse proxies from closing the connection.
Each queue is capped at 20 items so a stalled tab cannot cause unbounded memory
growth.

On browser tab close the `finally` block of the generator removes the queue from
the subscriber list cleanly.

### REST API

| Method | Path | Payload | Action |
|--------|------|---------|--------|
| `GET`  | `/` | — | Serve the web UI |
| `GET`  | `/api/status` | — | Snapshot of current status as JSON |
| `GET`  | `/api/events` | — | SSE stream of status updates |
| `POST` | `/api/command` | `{"action":"go","floor":0\|1}` | Move to floor |
| `POST` | `/api/command` | `{"action":"estop"}` | Immediate stop → FAULT state |
| `POST` | `/api/command` | `{"action":"home"}` | Trigger re-home |
| `POST` | `/api/command` | `{"action":"reset"}` | Reset fault and re-home |

`/api/command` publishes directly to `elevator/command` on the MQTT broker.
The `MQTTClient` already subscribed to that topic picks it up and calls the
appropriate FSM method — the web UI never touches the FSM or motor directly.

The `estop` action calls `fsm.on_safety_fault("web_estop")`, which has the same
effect as a physical NC switch opening: the motor stops immediately and the state
transitions to `FAULT`. Recovery requires a reset command.

### UI (`templates/index.html`)

Self-contained HTML/CSS/JS file — no external dependencies, no CDN calls. Works
on any phone browser on the home network without internet access.

| Element | Behaviour |
|---------|-----------|
| State badge | Colour-coded: green (idle), blue (moving), amber (homing), red (fault). Moving and homing badges pulse to draw attention. |
| Elevator shaft | CSS-positioned carriage div moves vertically. Position computed from `steps / TOTAL_STEPS`. Smooth `0.9s cubic-bezier` transition mirrors real carriage motion. |
| Floor buttons | Disabled automatically when state ≠ `idle` or already at that floor. |
| Emergency Stop | Always enabled. Sends `estop` action. |
| Re-home button | Enabled when state is `idle` or `fault`. |
| Reset button | Hidden unless state is `fault`. |
| Connection dot | Green when SSE is live, amber while reconnecting. SSE reconnects automatically on network drop. |

`TOTAL_STEPS` is injected into the page at render time via a Jinja2 template
variable so the shaft percentage is always correct without a separate config fetch.

## Kart Node Module Map

```
kart_node/main.py    KartNode
  │
  └── config.py      GPIO pin numbers, MQTT broker, topic names
```

### GPIO monitoring

```
GPIO 17 (DOOR_PIN)     — NC door switch, PUD_UP
GPIO 27 (PRESSURE_PIN) — NO pressure mat, PUD_DOWN

RPi.GPIO add_event_detect(BOTH) on each pin
  → _on_door_change()     → _publish_door("open"|"closed")
  → _on_pressure_change() → _publish_pressure(true|false)
```

Both topics are published **retained** (QoS 1) so the Pi 4B receives the current
kart state immediately on (re)connect without waiting for a pin change.

### Door safety logic (enforced on Pi 4B)

| Kart door state | Elevator state | Action |
|---|---|---|
| `open` | `idle` | `cmd_go()` blocked with log warning |
| `open` | `moving` | Motor stopped immediately, state → `fault` |
| `closed` | any | Normal operation |
| `unknown` | `idle` | `cmd_go()` allowed with log warning (degraded safety) |

## Mosquitto broker (Pi 4B)

Mosquitto runs as a system service on the Pi 4B and listens on port 1883.
No authentication is configured for the local network. If the elevator is
accessible from untrusted networks, add a password file:

```bash
sudo mosquitto_passwd -c /etc/mosquitto/passwd elevator
echo "listener 1883
allow_anonymous false
password_file /etc/mosquitto/passwd" | sudo tee /etc/mosquitto/conf.d/auth.conf
sudo systemctl restart mosquitto
```
